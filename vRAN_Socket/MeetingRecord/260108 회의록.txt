■ 주요 논의 내용

1. CPU-GPU 간 PCIe DMA 전송 [진의환 보고 / 교수님 논의]
--------------------------------------------------------------------------------
   • DMA 엔진(카피 엔진, DMA 컨트롤러)을 통해 PCIe 버스로 데이터 전송
   • PCIe가 받아들일 수 있는 최대 페이로드 사이즈로 순차 전송
   • PCIe 채널(레인)이 여러 개 있으므로 병렬 활용 가능
     - 현재는 CPU-GPU 외 다른 주변장치 사용 안 함 → PCIe 채널 전부 활용 가능
   • PCIe 버전(3/4/5/6)에 따라 최대 채널 수 상이
     - 현재 타이탄 GPU: PCIe 3 또는 4 사용 중

2. GPU 데이터 전송 실험 결과 [진의환 실험]
--------------------------------------------------------------------------------
   • 현재 구조
     - OAI 소켓 통신 유지 중 (C 파일 GPU 모드 전환은 시간 소요로 보류)
     - SINR/ESM 처리는 아직 미구현
     - GPU 전송, FFT 처리 시간 측정 완료
   
   • 테스트 환경
     - 타이탄 GPU, 싱글 링크, 1 TTI, 기지국 안테나 1개
     - 비동기 파이프라인/멀티 전송 미적용 (순차 처리)
   
   • 측정 결과 (CUDA 타임스탬프 사용)
     - H2D (Host to Device): 74 마이크로초
     - D2H (Device to Host): 62 마이크로초
     - 합계: 136 마이크로초 (1T 기준)
   
   • 예상 처리량
     - 1T ≈ 100 마이크로초 → 0.5ms(1 TTI)에 약 5개 가능
     - 현재 설정 기준 약 4T까지 가능 예상

3. 전송 효율화 방안 [교수님 제안 / 진의환 수행 예정]
--------------------------------------------------------------------------------
   • 큰 버퍼 한 번 전송 vs 작은 버퍼 여러 번 전송
     - 큰 버퍼: 오버헤드 1번 + 데이터 전송 시간
     - 작은 버퍼 N개: 오버헤드 N번 + 데이터 전송 시간
     → 큰 버퍼 한 번 전송이 오버헤드 최소화로 효율적
     → 100T 전송해도 200 마이크로초 이내 가능 예상 (선형 증가 아님)
   
   • 버퍼 설계 방향
     - 한 기지국 내 여러 TR: 각 TR 버퍼를 contiguous하게 배치
     - 여러 기지국/단말: 버퍼도 물리적으로 연속 배치
     - 인스턴스별 버퍼 어드레스를 오프셋으로 지정
       (예: 인스턴스 N번 → 버퍼 시작 주소 + N × 버퍼 크기)
     - 모든 인스턴스 버퍼 쓰기 완료 후 DMA 한 번에 전송

   • PCIe 양방향 동시 전송 가능 여부 확인 필요
     - 동시 가능 → PCIe 채널 문제 아님
     - 동시 불가 → GPU DMA 컨트롤러 병목
     - PCIe는 양방향이므로 이론상 동시 전송 가능

4. OAI 시뮬레이터 구조 확인 [진의환(채널) / 강민수(단말)]
--------------------------------------------------------------------------------
   • 프로그램 구성 (모두 별도 실행 파일)
     - 기지국 프로그램 (gNB)
     - 채널 프로그램 (진의환 개발 → 강민수 커스텀 사용)
     - 단말 프로그램 (UE)
   
   • 연결 방식
     - 기지국 ↔ 채널: 소켓 통신
     - 채널 ↔ 단말: 소켓 통신
     - 모두 소켓으로 연결, 별도 프로세스로 실행
   
   • 다중 단말 실행 방법 (강민수 경험)
     - IMSI를 다르게 설정
     - 별개 터미널에서 복수 실행
   
   • 다중 기지국 실행
     - 포트 분리 시 이론상 가능 (미검증)
     - 컨테이너 여러 개 또는 한 컨테이너 내 여러 인스턴스 가능

5. 다중 인스턴스 설계 [교수님 제안]
--------------------------------------------------------------------------------
   • 컨테이너 시나리오
     - 컨테이너 100개 → 각각 메모리 할당 → DMA 100개 리퀘스트
     - 비효율적: 각각 synchronous하게 처리됨
   
   • 효율적 설계 방안
     - 한 컨테이너 내 여러 인스턴스 실행
     - 인스턴스별 버퍼를 contiguous하게 미리 지정
     - 모든 인스턴스 버퍼 쓰기 완료 후 한 명이 DMA 명령 1회 전송
     → 큰 버퍼 한 통으로 전송 가능

6. 시스템 설계 방향 [교수님 제안]
--------------------------------------------------------------------------------
   • 한 TTI 기준 최대 처리량 결정 요소
     - CPU-GPU 전송 속도
     - GPU 내부 처리 속도
     - OAI 기지국/단말 처리 속도
     → 이 중 minimum이 보틀넥

   • 설계 원칙
     - CPU 1개 + GPU 1개 쌍에서 최대 TR 수 결정
     - 최대 TR 수 × 기지국 수 + 최대 TR 수 × 단말 수 맞춰서 설계
     - 5G 동기화 기준, propagation 딜레이 감안
   
   • 데이터 흐름
     - 업링크: 모든 단말 버퍼 쓰기 완료 → GPU 한 방에 전송 → 처리 → 기지국
     - 다운링크: 기지국 → GPU 처리 → 모든 단말에 한 방에 전송
   
   • 스케일러블 설계
     - 처음에 스케일러블하게 구현
     - 실제 사용 시 한 기지국 + 한 단말로 줄여서 테스트
     - 이후 점진적 확장

■ 개인별 연구 방향

1. 진의환
--------------------------------------------------------------------------------
   • 에이징 프리딕터 (Aging Predictor)
     - Before: 프리딕션 없음
     - After: 프리딕션 적용
     - 모빌리티/트래픽 시나리오 세팅
     - 클라이언트까지 붙여서 성능 관측
   • OSM 기반 네모 트윈에서 채널 데이터 활용

2. 피민지
--------------------------------------------------------------------------------
   • 주파수 소요량 모델
     - 스케줄링 디펜던트
     - 싱글 셀 기준
     - 준수와 인디펜던트하게 진행 (같은 주제지만 별개로)
   
   • 간섭 모형
     - 멀티셀 환경 필요
     - 인터피어런스 제너레이터 기지국/단말 배치
     - 셀별로 돌아다니며 간섭 수집
     - 간섭 데이터 통계 분석 및 활용 방안 탐색
     - TX/RX 시나리오별 간섭 정보 수집
   
   • 주파수 공유 시스템 전체 틀 구축 목표
   
   • 다음 주까지 연구 주제 구체화하여 제출
     - 시뮬레이터로 뭘 할 수 있는지 파악 후 결정
     - 스케줄러/상위 레이어 관심 시 gNB 개발은 패스 가능
   
   * 참고: 시뮬레이터 실험 결과를 논문에 직접 넣는 건 아니지만,
          이 경험이 이후 논문 작성에 큰 도움이 됨
